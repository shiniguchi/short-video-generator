  <action>
Create app/services/landing_page/copy_generator.py that generates LP copy using the existing LLM provider infrastructure.

Key functions:

1. `def generate_lp_copy(product_idea: str, target_audience: str, research_result: LPResearchResult, formula: str = "PAS") -> LandingPageCopy`:
   - Main entry point. Uses get_llm_provider() to generate structured copy.
   - Builds a detailed prompt that includes:
     a. The product idea and target audience
     b. Research context: top headline patterns, CTA styles, section trends from research_result
     c. The specific copywriting formula to follow (PAS default, AIDA alternative)
     d. Explicit tone instruction: "Conversational, friendly, direct, second-person ('you')"
     e. Social proof instruction: "Use implied proof only -- soft signals like 'Join X others' or 'Built by the team behind...' -- NO fake testimonials or fabricated stats" (per user decision)
     f. Output structure matching LandingPageCopy schema
   - Uses llm.generate_structured(prompt, LandingPageCopy, system_prompt=...) for structured output.
   - System prompt should establish the AI as an expert landing page copywriter.

2. Formula prompt templates (private constants):
   - `_PAS_PROMPT`: Problem-Agitate-Solution structure. Headline = Problem + Hook, Subheadline = Agitate emotional cost, Benefits = Solution details, CTA = Clear action.
   - `_AIDA_PROMPT`: Attention-Interest-Desire-Action. Headline = Attention grabber, Subheadline = Build interest, Benefits = Create desire, CTA = Direct action.
   - Each template has placeholders for {product_idea}, {target_audience}, {research_context}.

3. `def _format_research_context(research_result: LPResearchResult) -> str`:
   - Formats research patterns into a readable string for the LLM prompt.
   - Includes: top 3 headline examples, most common CTA text patterns, section order trends, video placement trend.

4. `def get_mock_copy(product_idea: str) -> LandingPageCopy`:
   - Returns realistic mock copy for development without LLM API calls.
   - Uses the product_idea in the headline for realism.
   - Follow project mock-first pattern.

5. `def generate_copy(product_idea: str, target_audience: str, research_result: LPResearchResult, use_mock: bool = False) -> LandingPageCopy`:
   - Top-level dispatcher: if use_mock, return mock copy. Otherwise call generate_lp_copy.
   - This is what the pipeline calls.

Use `from typing import Optional, List` for Python 3.9 compat. Import LandingPageCopy and LPResearchResult from app.schemas. Use logging.

  <action>
Create the Jinja2 template system with modular, Phase-15-ready section components.

**1. base.html.j2** — Base layout:
- `<!DOCTYPE html>`, `<html lang="en">`
- `<head>`: charset UTF-8, `<meta name="viewport" content="width=device-width, initial-scale=1.0">` (CRITICAL for mobile), `<title>{{ meta_title }}</title>`, `<meta name="description" content="{{ meta_description }}">`
- `<style>{{ inline_css }}</style>` — all CSS inlined (no external sheets)
- `<body>`: `{% for section in sections %}{{ section }}{% endfor %}`
- CSS reset block (*, box-sizing, margin/padding 0), body font-family (system-ui stack), base typography
- CSS custom properties using color scheme: `--color-primary`, `--color-secondary`, `--color-accent`, `--color-bg`, `--color-text`

**2. hero.html.j2** — Hero section:
- `<section class="hero" id="hero" data-section="hero">` (data-section attribute for Phase 15 targeting)
- h1 headline, p subheadline
- Video embed: `{% if video_url %}<video autoplay muted loop playsinline poster="{{ hero_image }}"><source src="{{ video_url }}" type="video/mp4"></video>{% elif hero_image %}<img src="{{ hero_image }}" alt="{{ headline }}" loading="eager">{% endif %}`
- No GIFs (per user decision — performance constraint)
- CTA button linking to #waitlist
- Mobile-first CSS: 2rem padding mobile, 4rem tablet (>=768px), 6rem desktop (>=1024px)
- CTA: min-height 48px, min-width 48px, padding 14px 28px, font-size 18px (thumb-friendly per LP-04)
- Hero video: width 100%, max-width 720px, border-radius 8px, margin auto

**3. benefits.html.j2** — Benefits section:
- `<section class="benefits" id="benefits" data-section="benefits">`
- `{% for benefit in benefits %}` loop rendering 3 benefit cards
- Each card: emoji icon (from benefit.icon_emoji), h3 title, p description
- Mobile: single column, stacked cards
- Tablet+: 3-column grid (`display: grid; grid-template-columns: repeat(3, 1fr)`)
- Cards: padding, border-radius, subtle shadow, centered text

**4. waitlist.html.j2** — Waitlist form:
- `<section class="waitlist" id="waitlist" data-section="waitlist">`
- Heading: "Join the Waitlist" or CTA-aligned text
- `<form id="waitlist-form">` (no action URL — Phase 16 handles backend)
  - Email input: `type="email"`, required, placeholder, min-height 48px, font-size 16px (prevents iOS zoom)
  - Honeypot field: `<input type="text" name="website_url" id="hp_field" class="hp-field" tabindex="-1" autocomplete="off">` with label
  - Hidden timestamp: `<input type="hidden" name="form_ts" id="form_ts">`
  - Submit button: min-height 48px, padding 14px 28px, uses accent color
- Honeypot CSS: `.hp-field, .hp-label { position: absolute; left: -9999px; width: 1px; height: 1px; opacity: 0; pointer-events: none; }` (NOT display:none — bots detect that, per research pitfall #4)
- JavaScript: set timestamp on load, validate on submit (reject if honeypot filled OR submitted < 2 seconds)
- Form submit handler: `e.preventDefault()` then show success message inline (no backend yet — Phase 16). Display "Thanks! You're on the list." message.
- Social proof text below form: `{{ social_proof_text }}`

**5. footer.html.j2** — Footer:
- `<section class="footer" id="footer" data-section="footer">`
- `{{ footer_text }}` — copyright, minimal branding
- Simple, clean, small text

**6. template_builder.py** — Template rendering engine:
- Uses `jinja2.Environment` with `FileSystemLoader` pointing to templates directory (use `Path(__file__).parent / "templates"` for the path)
- `def render_section(section_name: str, context: dict) -> str`: Renders a single section template. Loads from `sections/{section_name}.html.j2`.
- `def build_landing_page(copy: LandingPageCopy, color_scheme: ColorScheme, video_url: str = None, hero_image: str = None, sections_order: List[str] = None) -> str`:
  - Default sections_order: ["hero", "benefits", "waitlist", "footer"] (per user decision: lean default)
  - Renders each section with appropriate context
  - Builds CSS string with color custom properties from color_scheme
  - Collects all section CSS (each section template includes its own `<style>` block in a Jinja2 block)
  - Renders base.html.j2 with inline_css and sections list
  - Returns complete HTML string
- `def get_section_list() -> List[str]`: Returns available section names (for Phase 15 to query).

IMPORTANT: Each section template must be self-contained with its own styles using data-section selectors or BEM-namespaced classes (e.g., .hero__, .benefits__). This enables Phase 15 to replace individual sections without breaking others. Each section's CSS should be within the section's template, collected by the builder.
